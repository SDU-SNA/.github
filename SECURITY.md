## 🐛 报告安全漏洞

我们非常重视安全问题。如果你发现了安全漏洞,请遵循以下步骤:

### 请勿公开披露

**请不要**在公开的 Issue 中报告安全漏洞!这可能会让用户面临风险。

### 如何报告

1. **发送邮件**
   - 发送至: [安全团队邮箱 - 待添加]
   - 主题: `[SECURITY] 简短描述`

2. **包含以下信息**
   - 漏洞类型(如 XSS、SQL 注入、权限提升等)
   - 受影响的文件或组件
   - 重现步骤(越详细越好)
   - 潜在影响
   - 可能的修复建议(如果有)

3. **示例报告格式**
   ```
   漏洞类型: SQL 注入
   
   受影响组件: 
   - 文件: src/database/query.py
   - 函数: execute_user_query()
   
   重现步骤:
   1. 访问 /search 页面
   2. 在搜索框输入: ' OR '1'='1
   3. 观察返回所有用户数据
   
   潜在影响:
   - 未授权访问数据库
   - 可能泄露用户敏感信息
   
   建议修复:
   - 使用参数化查询
   - 添加输入验证
   ```

## ⏱️ 响应时间

- **初步确认**: 48 小时内
- **详细评估**: 7 天内
- **修复发布**: 根据严重程度,14-30 天

## 🎯 漏洞严重程度分级

### 严重 (Critical)
- 远程代码执行
- 未授权的数据访问
- 权限提升到管理员

### 高 (High)
- 敏感信息泄露
- 身份验证绕过
- SQL 注入

### 中 (Medium)
- XSS 攻击
- CSRF 攻击
- 信息泄露(非敏感)

### 低 (Low)
- 配置问题
- 最佳实践违反

## 🛡️ 安全最佳实践

### 对于贡献者

- 不要在代码中硬编码密码、API 密钥或其他敏感信息
- 使用环境变量存储配置
- 定期更新依赖项
- 遵循最小权限原则
- 对用户输入进行验证和清理

### 对于用户

- 保持软件更新到最新版本
- 使用强密码
- 不要分享你的凭证
- 定期检查访问日志
- 及时报告可疑活动

## 📋 安全检查清单

在提交代码前,请确认:

- [ ] 没有硬编码的敏感信息
- [ ] 所有用户输入都经过验证
- [ ] 使用了参数化查询(如适用)
- [ ] 实现了适当的访问控制
- [ ] 错误信息不泄露敏感信息
- [ ] 依赖项没有已知的安全漏洞
