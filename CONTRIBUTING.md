# 贡献指南

感谢你对 SDU-SNA 项目的关注!我们欢迎所有形式的贡献。

## 🎯 贡献方式

### 1. 报告问题 (Issue)

如果你发现了 bug 或有新功能建议:

- 先搜索现有 Issue,避免重复
- 使用 Issue 模板填写详细信息
- 提供清晰的复现步骤(如果是 bug)
- 附上相关的截图或日志

### 2. 提交代码 (Pull Request)

#### 开发流程

1. **Fork 项目**
   ```bash
   # 克隆你 fork 的仓库
   git clone https://github.com/你的用户名/项目名.git
   cd 项目名
   ```

2. **创建分支**
   ```bash
   # 从 main 分支创建新分支
   git checkout -b feature/你的功能名
   # 或
   git checkout -b fix/bug描述
   ```

3. **编写代码**
   - 遵循项目的代码规范
   - 添加必要的注释
   - 确保代码可以正常运行

4. **提交更改**
   ```bash
   git add .
   git commit -m "feat: 添加某某功能"
   ```

5. **推送分支**
   ```bash
   git push origin feature/你的功能名
   ```

6. **创建 Pull Request**
   - 在 GitHub 上创建 PR
   - 填写 PR 模板
   - 等待代码审查

## 📝 代码规范

### Commit 提交原则

#### 核心原则

1. **清晰明确 (Clear and Concise)**
   - 提交信息应该清楚地说明"做了什么"以及"为什么这么做"
   - 避免模糊不清或过于宽泛的描述
   - 让其他开发者能够快速理解变更内容

2. **原子性 (Atomic)**
   - 每次提交应该只包含一个逻辑上的变更
   - 修复一个 bug 和添加一个新功能应该分成两次提交
   - 便于代码审查、回滚和问题排查
   - 一个提交应该是可以独立理解和测试的

3. **格式化 (Structured)**
   - 采用统一的格式，方便工具解析和生成 CHANGELOG
   - 遵循约定式提交规范 (Conventional Commits)
   - 保持团队提交历史的一致性

#### Commit 信息格式

```
<类型>[可选的作用域]: <描述>

[可选的正文]

[可选的脚注]
```

**格式说明:**

- **类型**: 必填,说明提交的类别
- **作用域**: 可选,说明影响的范围(如模块名、文件名)
- **描述**: 必填,简短说明(建议不超过 50 字符)
- **正文**: 可选,详细说明变更的动机和实现细节
- **脚注**: 可选,关联 Issue 或说明破坏性变更

#### 类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | feat(auth): 添加用户登录功能 |
| `fix` | 修复 bug | fix(api): 修复数据获取失败的问题 |
| `docs` | 文档更新 | docs(readme): 更新安装说明 |
| `style` | 代码格式调整(不影响功能) | style: 统一代码缩进为 2 空格 |
| `refactor` | 重构代码(不修复 bug 也不添加功能) | refactor(utils): 优化日期处理函数 |
| `perf` | 性能优化 | perf(query): 优化数据库查询性能 |
| `test` | 添加或修改测试 | test(auth): 添加登录功能单元测试 |
| `build` | 构建系统或外部依赖变更 | build: 升级 webpack 到 5.0 |
| `ci` | CI 配置文件和脚本变更 | ci: 添加 GitHub Actions 工作流 |
| `chore` | 其他不修改源码的变更 | chore: 更新 .gitignore |
| `revert` | 回滚之前的提交 | revert: 回滚 feat(auth) 提交 |

#### 提交示例

**好的提交示例 ✅**

```bash
# 简单的功能添加
feat(auth): 添加用户登录功能

# 带详细说明的提交
feat(auth): 添加用户登录功能

实现了基于用户名密码的登录系统:
- 支持用户名/邮箱登录
- 添加记住密码选项
- 集成图形验证码验证
- 实现登录失败次数限制

Closes #123

# Bug 修复
fix(api): 修复用户数据获取失败的问题

当用户 ID 包含特殊字符时,API 请求会失败。
现在对 ID 进行 URL 编码处理。

Fixes #456

# 破坏性变更
feat(api)!: 重构用户 API 接口

BREAKING CHANGE: 用户 API 端点从 /user 改为 /api/v2/users
需要更新所有调用此接口的代码。

# 多个 Issue
fix(ui): 修复按钮样式问题

Closes #123, #456, #789
```

**不好的提交示例 ❌**

```bash
# 太模糊
update code

# 太宽泛,包含多个不相关的变更
feat: 添加登录功能,修复样式问题,更新文档

# 缺少类型
添加用户登录功能

# 描述不清晰
fix: 修复 bug

# 混合多个逻辑变更
feat: 添加登录和注册功能
```

#### 提交最佳实践

1. **使用祈使句**: "添加功能" 而不是 "添加了功能" 或 "添加功能中"
2. **首字母小写**: 描述部分首字母小写(中文无此要求)
3. **不加句号**: 描述结尾不需要句号
4. **正文换行**: 正文每行不超过 72 字符
5. **说明原因**: 在正文中解释"为什么"而不仅仅是"做了什么"
6. **关联 Issue**: 使用 `Closes #123` 或 `Fixes #456` 自动关闭 Issue

#### 提交前检查

- [ ] 提交是否只包含一个逻辑变更?
- [ ] 提交信息是否清晰描述了变更内容?
- [ ] 是否使用了正确的提交类型?
- [ ] 是否关联了相关的 Issue?
- [ ] 代码是否已经测试通过?

### 代码风格

- **Python**: 遵循 PEP 8 规范
- **JavaScript/TypeScript**: 使用 ESLint 配置
- **其他语言**: 参考各项目的 `.editorconfig` 文件

### 命名规范

- 变量名: 使用有意义的名称,避免单字母变量(除循环外)
- 函数名: 动词开头,清晰表达功能
- 类名: 使用大驼峰命名法
- 常量: 全大写,下划线分隔

## 🧪 测试要求

- 新功能需要添加相应的测试用例
- 确保所有测试通过后再提交 PR
- 测试覆盖率应保持在合理水平

```bash
# 运行测试
npm test
# 或
pytest
```

## 📋 Pull Request 检查清单

提交 PR 前请确认:

- [ ] 代码遵循项目规范
- [ ] 已添加必要的注释和文档
- [ ] 所有测试通过
- [ ] 没有引入新的警告或错误
- [ ] PR 描述清晰,说明了改动内容
- [ ] 已关联相关的 Issue
